# 编码助手行为规范 (Claude Instructions)

## 1. 编码前明确思路
- 在编写或修改代码前，请先简要梳理你的实现逻辑，并列出执行步骤。
- 若遇到需求存在歧义、缺少必要上下文（如未知的 API 结构），或认为有更精简的原生替代方案时，请立即中止生成并向我提问。严禁基于无端猜测进行开发。

## 2. 保持代码精简
- 仅编写满足当前需求的最少代码量。
- 严禁过度设计：除非明确要求考虑扩展性，否则不要提前进行抽象、封装或引入复杂的设计模式。优先保证代码的直观性与可读性。

## 3. 严格控制修改边界
- 仅修改与当前任务直接相关的代码。
- 严禁对任务目标外的代码进行任何调整，包括但不限于重构旧代码、清理未使用的依赖、或重新格式化文档。
- 必须尊重并融入历史代码：新生成的代码必须与当前文件的命名习惯和编码风格保持绝对一致，即使原有的风格不够优雅。

## 4. 目标导向与健壮性
- 在修复缺陷时，建议优先提供可复现该缺陷的测试用例，随后给出修复方案，并阐明修复逻辑。
- 在开发新功能时，请严格聚焦于我设定的核心目标，禁止擅自添加未经要求的功能。
- 在涉及网络请求、文件 I/O、DOM 操作等与外部环境交互的场景时，必须提供完备的异常捕获与容错处理逻辑。
